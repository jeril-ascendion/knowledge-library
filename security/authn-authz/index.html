Authentication & Authorization

Proving identity and enforcing access rights across every system interaction

TOGAF ADM NIST CSF ISO 27001 AWS Well-Arch Google SRE AI-Native
💡
In Plain English

Authentication is the lock on your front door — it checks if you are who you say you are. Authorization is the list of which rooms you're allowed to enter once you're inside. Getting either of these wrong is the most common cause of data breaches.

📈
Business Value

Identity and access misconfigurations caused 15% of all reported breaches in 2024 (Verizon DBIR). OAuth 2.0 + OIDC is the standard adopted by Google, Microsoft, AWS, and every major bank. A properly designed identity architecture reduces breach blast radius, simplifies compliance evidence collection, and enables zero-trust enforcement.

📖 Detailed Explanation

Authentication and authorization are the most security-critical components of any application architecture. They are also among the most commonly misimplemented — not because the concepts are obscure, but because the implementation requires navigating a complex landscape of protocols, token types, flow variations, and integration patterns.

OAuth 2.0 and OIDC are the industry-standard protocols for authentication and authorization in modern web and mobile applications. OAuth 2.0 is an authorization framework — it defines how an application can obtain limited access to a user's data on another service (e.g., a third-party app reading your Google Calendar). OpenID Connect (OIDC) is an identity layer built on top of OAuth 2.0 — it adds standardized ID tokens (JWTs) that assert the user's identity. Together, they handle both "what can this app do?" (OAuth 2.0) and "who is this user?" (OIDC).

OAuth 2.0 Flows: The right flow depends on the client type. Authorization Code + PKCE is for web SPAs and mobile apps — public clients that cannot keep a secret. PKCE (Proof Key for Code Exchange) prevents authorization code interception attacks by requiring the client to prove it initiated the request. Client Credentials is for machine-to-machine (M2M) communication — a service authenticating to another service without a human user in the flow. Device Code is for devices without browsers (IoT, CLI tools). The mistake most teams make: using Client Credentials for user-facing flows because it's simpler. This skips OIDC entirely and creates non-auditable user sessions.

JWT Token Anatomy: A JWT (JSON Web Token) consists of three base64-encoded parts: Header (algorithm, token type) + Payload (claims: sub, iss, aud, exp, iat, plus custom claims) + Signature (HMAC or RSA). The signature proves the token was issued by a trusted authority and hasn't been tampered with. Key security decisions: token lifetime (access tokens: 5–60 minutes; refresh tokens: days to weeks with rotation), signing algorithm (RS256/ES256 for asymmetric keys, enabling public key verification by any service without sharing the signing key), and audience validation (each service must validate that the `aud` claim includes its own identifier — otherwise a token for Service A can be reused for Service B).

Fine-Grained Authorization: RBAC (Role-Based Access Control) maps users to roles (admin, reader, editor) and roles to permissions. It's simple and widely supported but breaks down at scale — organizations with hundreds of roles, roles that are never reviewed, and "role explosion" where a new role is created for every edge case. ABAC (Attribute-Based Access Control) evaluates policies based on subject attributes (user.department = Finance), resource attributes (document.classification = Confidential), and environmental attributes (request.time, request.location). OPA (Open Policy Agent) implements ABAC as code — policies expressed in Rego, version-controlled, tested in CI, deployed as a sidecar to each service.

Session Management: Stateless sessions using JWTs are the modern standard — the token carries all claims, no server-side session store needed. But statelessness makes revocation hard: a revoked token remains valid until expiration. Solutions: short access token lifetimes (15 minutes) with refresh tokens that are revoked centrally; token introspection (service calls authorization server to validate token on each request); token binding (binding the token to a specific TLS session). For high-security contexts (banking, government), short lifetimes + introspection is the appropriate choice.

Zero Trust Authentication: In a Zero Trust model, network location provides no implicit trust. A request from inside the corporate network is as untrusted as one from the public internet. Every service-to-service call must authenticate via mTLS or a service identity token (SPIFFE/SPIRE). Every user request must authenticate via OIDC regardless of network origin. Ambient trust based on IP address is eliminated. This is the architectural model mandated by NIST SP 800-207 and aligned with Google's BeyondCorp implementation.

🌟 Core Principles

1
Authentication and Authorization are Separate Concerns

Authentication establishes who you are. Authorization determines what you can do. Never conflate the two in code — authenticate at the API gateway, authorize within each service using a policy engine.

2
Token Lifetimes Must Match Risk Level

Access tokens for financial operations: 5 minutes. Standard API access tokens: 15–60 minutes. Refresh tokens: 7 days with single-use rotation. Longer lifetimes increase breach blast radius; shorter lifetimes increase token refresh overhead.

3
Centralize Identity, Federate Authorization

One Identity Provider for authentication. Per-service authorization policy enforcement. Never replicate user management logic across services — it diverges immediately.

4
PKCE is Mandatory for Public Clients

Any client that cannot securely store a client secret (SPAs, mobile apps) must use PKCE. Without PKCE, authorization codes can be intercepted and exchanged by an attacker.

5
Never Implement Custom Authentication

Use battle-tested libraries and standards. Custom JWT validation routines contain subtle vulnerabilities (algorithm confusion attacks, missing audience validation, improper expiry checking) that take years to surface and can be catastrophic.

⚙️ Implementation Steps

1

Select and Deploy an Identity Provider

Choose an IdP appropriate to your context: Keycloak (self-hosted, open source, strong for regulated environments), AWS Cognito (managed, tight AWS integration), Azure AD B2C (enterprise identity federation), or Okta (enterprise SSO). Document the choice in an ADR with alternatives considered.

2

Design OAuth 2.0 Flows per Client Type

Map each client type to its flow: SPA → Authorization Code + PKCE. Mobile → Authorization Code + PKCE. Server-side web → Authorization Code. Service-to-service → Client Credentials. CLI → Device Code. Document all flows with sequence diagrams.

3

Implement Fine-Grained Authorization with OPA

Deploy OPA as a sidecar or centralized policy service. Write authorization policies in Rego. Express policies as: subject (who) + resource (what) + action (how) + context (when/where). Test policies with OPA's built-in test framework. Version-control policies alongside application code.

4

Configure Token Lifetimes and Rotation

Access tokens: 15 minutes default (5 minutes for financial operations). Refresh tokens: 7 days with single-use rotation (old refresh token revoked when a new one is issued). Implement refresh token rotation to detect token theft.

5

Implement MFA for High-Value Operations

MFA must be required for: all administrative access, financial transaction authorization, data export, and first login from a new device. Support authenticator apps (TOTP), WebAuthn/Passkeys (most phishing-resistant), and SMS OTP (last resort — vulnerable to SIM swap attacks).

◈ Recommended Patterns

✦ OAuth 2.0 + OIDC with PKCE

The industry standard for user authentication in web and mobile applications. Delegates identity management to a specialized IdP, supports MFA, enables single sign-on, and produces standardized JWT tokens consumable by any service.

✦ Policy-as-Code with OPA

Authorization policies expressed in Rego, stored in Git, tested in CI, deployed as a sidecar to each service. Enables consistent, auditable, version-controlled authorization logic across a polyglot microservices landscape.

✦ Token Introspection for Revocation

Instead of trusting JWT expiry alone, services call the authorization server's introspection endpoint to validate tokens in real time. Enables immediate revocation at the cost of per-request latency (mitigated with local caching of introspection results).

✦ Just-In-Time (JIT) Privileged Access

Administrative access is provisioned on-demand for a specific operation and time window (e.g., 2-hour production database access). Automated provisioning via PIM (Azure), IAM Access Analyzer, or CyberArk. Access is automatically revoked when the window expires.

⛔ Anti-Patterns to Avoid

⛔ Long-Lived Static API Keys

API keys that never expire, are shared across environments, and are stored in configuration files. A single key leak provides indefinite access. Replace with short-lived tokens issued by an IdP, or with workload identity (IAM roles, managed identities) for service-to-service authentication.

⛔ Role Explosion

Creating a new RBAC role for every combination of permissions requested by every team. Organizations with 500+ roles have no operational visibility into effective permissions. ABAC with OPA scales where RBAC cannot.

⛔ Ambient Network Trust

Trusting requests from internal networks without authentication. 'It's behind our firewall' is not an access control. Internal lateral movement in breaches exploits exactly this assumption. Implement Zero Trust: authenticate every request regardless of origin.

🤖 AI Augmentation Extensions

🤖 Anomalous Access Detection

ML models trained on historical access logs learn normal patterns per user and service account. Deviations — unusual access times, unfamiliar resource types, high-volume access — trigger real-time security alerts.

⚡ Tune anomaly detection with 90 days of baseline data before enabling automated account suspension. False positives from under-tuned models erode trust in the system.
🤖 AI-Generated OPA Policy Drafts

LLM agents translate natural-language access control requirements ('Finance users can read but not export P&L reports outside business hours') into Rego policy drafts with test cases.

⚡ All AI-generated Rego policies must be reviewed by a security engineer before deployment. LLMs can generate syntactically correct but semantically incorrect policies.
