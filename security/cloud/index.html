Cloud Security

Cloud security posture management (CSPM), CWPP, IAM least privilege, and guardrails-as-code.

TOGAF ADM NIST CSF ISO 27001 AWS Well-Arch Google SRE AI-Native
💡
In Plain English

Cloud Security is a core discipline within Security Architecture. It defines how technology systems should be designed, implemented, and governed to achieve reliable, secure, and maintainable outcomes that serve both technical teams and business stakeholders.

📈
Business Value

Applying Cloud Security standards reduces system failures, accelerates delivery, and provides the governance evidence required by enterprise clients, regulators like BSP, and certification bodies like ISO. Top technology companies (Google, Microsoft, Amazon) treat these standards as competitive differentiators, not compliance overhead.

📖 Detailed Explanation

Security architecture designs the security controls, policies, and technical mechanisms that protect systems, data, and operations from threats. It applies defense-in-depth across network, application, data, and identity layers.

Industry Context: Security tooling: SIEM (Splunk, Microsoft Sentinel), CSPM (Prisma Cloud, AWS Security Hub), DAST (Burp Suite), SAST (Checkmarx, Semgrep).

Relevance to Philippine Financial Services: Organizations operating under BSP supervision must demonstrate mature security architecture practices during technology examinations. The BSP Technology Supervision Group evaluates documentation quality, process maturity, and evidence of systematic practice — all of which are addressed by the standards in this section.

Alignment to Global Standards: The practices documented here are aligned to frameworks used by Google, Amazon, Microsoft, and the world's leading consulting firms (McKinsey Digital, Deloitte Technology, Accenture Technology). They represent the current industry consensus on best practices rather than any single vendor's approach.

Engineering Perspective: For engineers, Cloud Security provides concrete patterns and anti-patterns that prevent common mistakes and accelerate development by providing proven solutions to recurring problems. Rather than rediscovering what doesn't work, teams can apply battle-tested approaches with known trade-offs.

Architecture Perspective: For architects, Cloud Security provides the design vocabulary, decision frameworks, and governance artifacts needed to make and communicate complex technical decisions clearly and consistently.

Business Perspective: For business stakeholders, Cloud Security provides assurance that technology investments are aligned to industry standards, reducing the risk of expensive rework, regulatory findings, and system failures that impact customers and revenue.

🌎 Real-World Examples

CrowdStrike — Zero Trust Implementation
Austin, USA · Cybersecurity · 23,000+ enterprise customers

CrowdStrike's Falcon platform enforces Zero Trust by treating every endpoint as potentially compromised regardless of network location. Device health score, user identity, behavioral signals, and real-time threat intelligence combine in every authorization decision. After analyzing the 2020 SolarWinds supply chain attack, CrowdStrike published a STRIDE threat model for supply chain attacks now used as a template by US CISA.

✓ Result: Detects 230,000+ attacks daily with < 0.01% false positive rate; median threat containment time: 1 minute from detection

ING Bank — Defense in Depth
Amsterdam, Netherlands · Retail Banking · 38 countries

ING's post-GDPR security overhaul implemented all 6 layers of defense in depth across 53,000 employees. FIDO2 hardware keys replaced SMS OTP for all staff — eliminating SIM swap attacks. All customer PII is tokenized using format-preserving encryption. OPA policies govern all service-to-service API authorization. The Dutch Central Bank (DNB) cited ING as a benchmark for operational resilience.

✓ Result: Phishing-based account compromises dropped 99.7% post-FIDO2; zero regulatory security findings in 4 consecutive DNB examinations

Stripe — API Security Reference
San Francisco, USA · Payments · 3M+ businesses

Stripe's API security architecture is an OWASP reference implementation. Every API key is scoped to specific permissions (RBAC), time-bounded, and geo-restricted. HMAC webhook signatures prevent request forgery. mTLS is required for all enterprise integrations. Their bug bounty program pays $30,000+ for critical API vulnerabilities — turning external researchers into a security layer.

✓ Result: Processes $817B annual payment volume with zero successful API key forgery attacks in production history

Target — Post-Breach Transformation
Minneapolis, USA · Retail · 1,800+ stores

After the 2013 breach (40M credit cards via HVAC vendor credentials), Target rebuilt their entire security architecture. STRIDE threat modeling is now mandatory for every vendor integration. Micro-segmentation isolates payment systems. Zero Trust replaced flat network trust for all 1,800+ suppliers. This transformation is a Harvard Business School case study.

✓ Result: Zero PCI DSS findings in 9 consecutive post-transformation audits; vendor security incidents reduced 87%

🌟 Core Principles

1
Intentional Design for Cloud Security

Every aspect of cloud security must be deliberately designed, not discovered after deployment. Document design decisions as ADRs with explicit rationale.

2
Consistency Across the Portfolio

Apply cloud security practices consistently across all systems. Inconsistent application creates governance blind spots and makes incident investigation unpredictable.

3
Alignment to Business Outcomes

Cloud Security practices must demonstrably contribute to business outcomes: reduced downtime, faster delivery, lower operational cost, or improved compliance posture.

4
Evidence-Based Quality Assessment

Quality of cloud security implementation must be measurable. Define specific metrics and collect evidence continuously — not only at audit or review time.

5
Continuous Evolution

Standards for cloud security evolve as technology and threat landscapes change. Schedule quarterly reviews of applicable standards and update practices accordingly.

⚙️ Implementation Steps

1

Current State Assessment

Document the current state of cloud security practice: what is implemented, what is missing, what is inconsistent across teams. Use the governance/scorecards section for a structured assessment framework.

2

Gap Analysis Against Standards

Compare current state against the standards in this section and applicable frameworks (NIST Cybersecurity Framework, OWASP ASVS). Prioritize gaps by business impact and remediation effort.

3

Design the Target State

Define the target cloud security state: which patterns will be adopted, which anti-patterns eliminated, which governance mechanisms introduced. Express as a time-bound roadmap.

4

Incremental Implementation

Implement cloud security improvements incrementally: pilot with one team or system, measure outcomes, refine the approach, then expand. Avoid big-bang transformations.

5

Validate and Iterate

Measure the impact of implemented changes against defined success criteria. Incorporate lessons learned into the practice standards. Contribute improvements back to this library.

◈ Recommended Patterns

✦ Reference Architecture Adoption

Start from an established reference architecture for cloud security rather than designing from scratch. Adapt to organizational context rather than rebuilding proven foundations.

✦ Pattern Library Contribution

When your team solves a recurring cloud security problem with a novel approach, document it as a pattern for the library. This compounds organizational knowledge over time.

✦ Fitness Function Testing

Encode cloud security standards as automated architectural fitness functions — tests that run in CI/CD and fail builds when standards are violated. This makes governance continuous rather than periodic.

⛔ Anti-Patterns to Avoid

⛔ Standards Theater

Documenting cloud security standards in architecture policies that no one reads and no one enforces. Standards without automated validation or governance gates are not operational standards.

⛔ Copy-Paste Architecture

Adopting another organization's cloud security patterns wholesale without adapting to organizational context, team capability, or regulatory environment. Always adapt; never just copy.
